# AGENTS.md

## Project Premise

このワークスペースは、公共団体向けCMSへのサイト移行作業と、移行時に行うアクセシビリティ修正作業を効率化するための検討・知識整理を行う場所である。

利用企業は、日本の地方自治体など公共団体を主な顧客とするCMSを開発・提供している。このCMSは400以上の顧客に利用されており、新規顧客の導入時には、旧CMSや旧サイトから自社CMSへコンテンツを移行する必要がある。

## Target Work

このプロジェクトで扱う作業は、主に次の2つである。

1. 移行作業
   - 旧CMSのDBから新CMSのDBへ直接データを移す作業ではない。
   - 旧サイト上に公開されているHTMLや関連ファイルから必要な情報を取り出す。
   - 取り出した情報を、自社CMSの管理画面から、新CMSのデータ構造・テンプレート・入力欄に合う形で登録し直す。
   - ページ本文だけでなく、画像、PDFなどのファイル、リンク、表、見出し、注記、連絡先表記なども対象になり得る。

2. アクセシビリティ修正作業
   - 移行作業のタイミングで、WCAG 2.0に合うようにアクセシビリティ上の問題を修正する。
   - 旧サイトのHTMLを単純に再現するのではなく、利用者にとって理解・操作しやすいコンテンツとして再構成する。
   - ただし、原文の意味、行政情報としての正確性、自治体ごとの表記方針を不用意に変えない。

## Current Manual Workflow

現状の移行作業とアクセシビリティ修正作業は、主に人手による確認・判断・CMS登録を中心に進めている。

### 移行作業

- 既存ページのページ一覧を作成する。
- ページ一覧をデザインテンプレートごとにカテゴライズする。
  - 例: 本体サイト、子育てサイト、観光サイトなど。
- サイトカテゴリーごとにCMSへページ情報を登録する。
- 移行対象として必要なのはコンテンツ部分であるため、HTMLのID、class、要素などを手がかりに必要な情報を絞り込む。
- 絞り込んだHTMLを、移行作業に必要なHTMLとしてCMSに登録する。

### アクセシビリティ修正作業

- 既存の移行・アクセシビリティ関連ナレッジは `a11y-migration-kb/` にある。
- 作業者は、移行・アクセシビリティ関連ナレッジに沿って、CMS上で全て目視確認しながら修正する。
- 作業者が修正を完了したら承認者に渡す。
- 承認者は再度目視で確認し、問題がなければそのページを完了とする。
- 作業記録は全てスプレッドシート上で管理し、証跡を残す。

## Content-Only Scope Boundary

本プロジェクトの主対象は、旧サイトHTML全体ではなく、CMSに登録する「コンテンツ部分」である。

- アクセシビリティ修正の基準は、コンテンツ部分に特化した `a11y-migration-kb/` を優先する。
- ヘッダー、グローバルナビゲーション、パンくず、サイト内検索、フッター、テンプレート共通部品、広告枠、解析タグ、CMS側レイアウト部品は、原則として移行対象コンテンツの修正範囲に含めない。
- 外部アクセシビリティ検査ツールがページ全体を対象に出した指摘は、そのまま修正候補にしない。
- ページ全体の検査結果は、少なくとも `content`、`old-site-template`、`new-cms-template`、`unknown` に分類し、`content` に該当するものだけを移行修正候補として扱う。
- 外部ツールを使う場合は、可能な限りコンテンツ部分のroot selector、抽出済みHTML断片、またはCMS登録後プレビュー内の本文領域にスコープして実行する。
- コンテンツ範囲の抽出条件、除外した領域、スコープ外として扱った指摘は、後から確認できるように記録する。

## miChecker Quality Signal

公共団体向け案件では、WCAGやJISの達成基準そのものよりも、総務省が提供する `miChecker` で違反・問題として指摘されるかどうかが、実務上の品質確認や検収判断に強く影響する場合がある。

- miCheckerは、JIS X 8341-3:2016に基づくウェブアクセシビリティ対応の取組を支援するための評価ツールとして扱う。
- miCheckerは `a11y-migration-kb/` を置き換える主基準ではなく、公共団体案件における重要な受け入れシグナル・品質ゲート候補として扱う。
- miCheckerで本文コンテンツ起因の明らかな問題が残る場合は、原則として完了扱いにしない。
- miCheckerの結果はページ全体検査になりやすいため、`content`、`old-site-template`、`new-cms-template`、`unknown` に分類して扱う。
- `old-site-template` や `new-cms-template` 起因の指摘は、移行HTML修正とは分離し、テンプレート課題またはスコープ外課題として記録する。
- miCheckerを使う場合は、実行日、miCheckerバージョン、検査対象URLまたはHTML、検査対象種別、指摘分類、対応結果、未解消理由を証跡として残す。

## Important Constraints

- 顧客は公共団体であり、情報の正確性、説明責任、公開前確認が重要である。
- 移行元サイトのHTMLは、構造・品質・表記ゆれ・アクセシビリティ対応状況が案件ごとに異なる。
- CMS管理画面への登録作業には、人手による判断や確認が残る。
- 旧サイトの見た目を完全再現することよりも、新CMSの構造に沿った適切な登録とアクセシビリティ改善を優先する。
- ルール化できる処理、AIで下書きできる処理、人間の確認が必要な処理、SVや顧客確認が必要な処理を分けて考える。
- 判断に迷う内容、行政情報の意味を変える可能性がある内容、自治体固有ルールに関わる内容は自己判断で確定しない。

## Efficiency Goals

効率化の検討では、単に作業時間を短縮するだけでなく、次の観点を同時に満たすことを目指す。

- 移行作業者が迷う回数を減らす。
- アクセシビリティ修正の品質を平準化する。
- 修正理由と判断根拠を後から説明できるようにする。
- CMS登録時の手戻りを減らす。
- 自動化できる作業と、人間が確認すべき作業を明確に分ける。
- 案件固有ルールをナレッジとして蓄積し、次の案件で再利用できるようにする。

## Agent Working Policy

このワークスペースで作業するエージェントは、次の方針に従う。

- 既存の知識ベースを優先して参照する。
- ルールや手順を追加する場合は、実作業者が判断に使える粒度で書く。
- アクセシビリティ修正案を出す場合は、修正前後の差分、修正理由、関連するWCAG観点を明示する。
- 機械的に変換できるものと、文脈判断が必要なものを混同しない。
- 行政情報の意味を変える可能性がある提案は、確認が必要な事項として扱う。
- 実装や自動化を検討する場合は、CMS管理画面への入力制約、既存テンプレート、移行管理シート、質問フローを前提にする。

## AI Accessibility Generation Policy

AIによるアクセシビリティ修正・HTML生成は、モデル単体の一般能力に依存せず、短い共通基本指示、部品別Skill、生成後レビュー、自動検証と人間確認の分離を組み合わせて扱う。

この方針は、Mark Fairchild の記事 "AI-Generated Accessibility: An Update - Frontier Models Still Fail, but Skills Change the Game"（2026-06-29参照）を、本プロジェクトの前提に合わせて採用したものである。

- プロジェクト共通の基本指示は短く保ち、全作業に常時適用する。詳細な判断手順は `a11y-migration-kb/` や部品別Skillへ分離する。
- table、iframe、画像alt、フォーム、見出し構造、リンク、テキスト表記など、失敗パターンが異なる部品は同じプロンプトで一括処理しない。部品別Skillとして、入力条件、変換方針、禁止事項、レビュー観点、証跡項目を分ける。
- 生成後は必ずレビュー工程を置く。AGENTが出したHTMLや修正案を、構造、意味保持、CMS入力制約、WCAG/JIS観点、miCheckerで指摘されやすい点に照らして確認する。
- 自動検証で判定できる項目と、人間が文脈判断すべき項目を混同しない。自動検証は候補分類と品質シグナル、人間確認は意味・行政情報・自治体方針・顧客確認事項の確定に使う。
- 「自動修正できたように見える」だけでは完了扱いにしない。修正前後、修正理由、残した要確認事項、検証結果を証跡に残す。

## Existing Knowledge Base

既存の移行・アクセシビリティ関連ナレッジは `a11y-migration-kb/` にある。

- `a11y-migration-kb/README.md`: ナレッジベース全体の概要
- `a11y-migration-kb/workflow/`: 移行作業・公開作業のフロー
- `a11y-migration-kb/rules/`: テキスト、リンク、画像、表、ファイル、HTML構造などの移行ルール
- `a11y-migration-kb/cms/`: CMS機能・操作に関する情報
- `a11y-migration-kb/reference/`: 移行管理シートや質問プロトコルなどの参照情報
- `a11y-migration-kb/wcag/`: WCAG/JISに関する根拠情報

## Current Discussion Focus

今後の検討テーマは、移行作業とアクセシビリティ修正作業を一体として効率化する方法である。具体的には、旧サイトHTMLの取得、構造解析、CMS入力形式への変換、アクセシビリティ修正候補の生成、作業者確認、質問・エスカレーション、移行管理シートへの記録までを、どのように分担・自動化・標準化できるかを検討する。
