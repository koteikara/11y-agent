# workstream.md

## Purpose

この文書は、移行作業とアクセシビリティ修正作業を効率化するためのゴールと想定ワークストリームを定義する。

効率化では、AGENTがLLMなどを活用しながら、既存の移行・アクセシビリティ関連ナレッジに沿って機械的に対応できる部分を先に処理する。そのうえで、作業者と承認者が確認・修正・承認する流れを残す。

## Assumptions

- AGENTは `a11y-migration-kb/` のナレッジを参照し、移行対象HTMLに対してアクセシビリティ修正候補を適用する。
- AGENTの出力は最終確定情報ではなく、作業者または承認者による確認対象である。
- 行政情報の意味を変える可能性がある修正、自治体固有ルールに関わる修正、判断根拠が不足する修正は、人間の確認対象として扱う。
- CMSへの登録前または登録中に、HTMLの抽出、不要要素の除去、構造整理、アクセシビリティ修正、確認記録の作成を行う。
- 作業記録と証跡は、スプレッドシートなど既存の管理方法と接続する。
- 公共団体案件では、miCheckerで指摘されるかどうかが実務上の品質判断に使われる可能性が高いため、miCheckerを受け入れシグナル・品質ゲート候補として扱う。
- Goal 2では、A11yc ACSのようにHTMLを入力し、問題箇所・理由・関連基準を明示しながら、作業者が確認したうえで最終HTMLを出力できる実行画面を有力な参考にする。

## Goal 1: Batch Accessibility Optimization Before CMS Registration

CMSに登録するHTMLを、事前にAGENTを通じて一括でアクセシビリティ対応し、最適化された状態でCMSへ登録するワークストリーム。

### Target Flow

1. 既存サイトからページ一覧を作成する。
2. ページ一覧をデザインテンプレートまたはサイトカテゴリーごとに分類する。
3. HTMLのID、class、要素などを手がかりに、移行に必要なコンテンツ部分を抽出する。
4. 抽出したHTMLに対して、AGENTが一括でアクセシビリティ修正を行う。
5. AGENTが修正後HTML、修正内容、判断根拠、要確認事項を出力する。
6. 可能であれば、抽出済みHTMLを検査用HTMLにラップし、miChecker相当の事前確認を行う。
7. 作業者が最適化済みHTMLをCMSへ登録し、必要に応じて確認・修正する。
8. CMS登録後のプレビューURLでmiChecker確認を行い、指摘を `content`、`old-site-template`、`new-cms-template`、`unknown` に分類する。
9. 承認者がCMS上で確認し、本文起因の明らかな問題がなければページ完了とする。
10. 作業記録、miChecker確認結果、分類、未解消理由をスプレッドシートに残す。

### Expected Benefits

- CMS登録前に多数ページをまとめて処理できる。
- 同種のアクセシビリティ修正を一括適用しやすい。
- 作業者はゼロから目視修正するのではなく、AGENTの修正結果を確認する作業に寄せられる。
- テンプレートやサイトカテゴリー単位で処理ルールを標準化しやすい。

### Key Risks

- 一括処理の誤変換が多数ページに広がる可能性がある。
- ページごとの文脈判断が必要な箇所を見落とす可能性がある。
- CMS登録後の見え方や入力欄制約により、事前最適化したHTMLの再修正が必要になる可能性がある。

## Goal 2: Step-by-Step Accessibility Optimization During CMS Registration

CMSに登録するタイミングで、作業者が1ページずつAGENTの支援を受けながらアクセシビリティ対応し、登録作業を進めるワークストリーム。

### Execution Screen Hypothesis

Goal 2の実行画面は、A11yc ACSを参考に、単なる検査結果表示ではなく「HTML入力から修正後HTML出力まで」を扱う画面として検討する。

- 作業者が移行対象のHTMLまたはCMS入力予定HTMLを貼り付ける。
- AGENTまたは検査エンジンが、問題箇所を本文中の位置、該当HTML、理由、関連する `a11y-migration-kb/` ルール、WCAG/JIS観点と一緒に表示する。
- 指摘ごとに、機械修正可能、作業者確認、承認者確認、SV/顧客確認、スコープ外を分類する。
- 作業者が候補を確認・採用・修正・却下できる。
- 採用結果を反映した最終HTMLを出力し、CMSに登録できる。
- 修正前HTML、修正後HTML、採用/却下理由、要確認事項、検査結果を証跡として残す。

### Candidate Review Interaction

修正候補の確認操作は、GitHubのコンフリクト解消に近い「候補ごとの採用/編集/却下」モデルを参考にする。

- 画面の主表示は、ソースコードだけでなくレンダリングされたHTMLを基本にする。
- 問題箇所をレンダリング結果上でハイライトし、対応するHTML断片、修正理由、関連ルールを横に表示する。
- 各候補に対して、作業者が `採用`、`編集して採用`、`却下`、`要確認` を選べる。
- `採用` はAGENT候補をそのまま最終HTMLへ反映する。
- `編集して採用` は、候補HTMLや文言を作業者が調整したうえで最終HTMLへ反映する。
- `却下` は、候補を反映せず、却下理由を証跡として残す。
- `要確認` は、作業者判断で確定せず、承認者、SV、顧客確認へ渡す。
- 操作後は、レンダリング結果と最終HTMLが同期して更新される。
- すべての候補が `採用`、`編集して採用`、`却下`、`要確認` のいずれかに分類されるまで、ページ作業を完了扱いにしない。

### Hosting Direction

Goal 2の初期開発では、実行画面をCloud Run上の独立Webアプリとしてホストする方針を第一候補にする。

- Cloud Run上で、HTML入力、レンダリングプレビュー、問題箇所表示、候補レビュー、最終HTML出力、証跡出力を行う。
- 初期PoCでは、CMS本体へ直接組み込まず、CMS登録予定HTMLを貼り付けて最終HTMLを得る画面として作る。
- 実案件HTMLを扱う段階では、無認証公開せず、社内認証、IAP、IAM、VPN、または既存SSO連携などでアクセス制限する。
- A11yc library、axe系検査、LLM/API呼び出しは、可能な限りサーバー側で実行し、APIキーや検査ロジックをクライアント側へ露出しない。
- 旧サイト由来HTMLのレンダリングは、sandbox付きiframeなどを使い、script実行や意図しない外部通信を避ける。
- miCheckerは初期Cloud Run構成へ直接組み込まず、CMS登録後プレビューの手動確認ゲートとして扱い、結果を証跡化する。
- 画面、候補形式、証跡形式が固まった後で、CMS管理画面への組み込みやブラウザ拡張補助を再検討する。

### Target Flow

1. 作業者が対象ページを選択する。
2. 作業者が移行対象HTMLまたはCMS入力予定HTMLを実行画面に入力する。
3. AGENTが対象HTMLから、移行に必要なコンテンツ部分を確認する。
4. AGENTが `a11y-migration-kb/` に沿って、ページ内の問題箇所と修正候補をステップごとに提示する。
5. 作業者はレンダリングされたHTML上の問題箇所を確認しながら、候補の採用、編集して採用、却下、要確認化を行う。
6. AGENTは採用結果を反映した最終HTML、修正済み箇所、未確認箇所、要質問箇所を整理する。
7. 作業者が最終HTMLをCMSへ登録する。
8. 作業者がCMS登録後のプレビューURLでmiChecker確認を行い、指摘が本文起因かテンプレート起因かを分類する。
9. 作業者がページ単位の作業を完了させる。
10. 承認者がCMS上で確認し、本文起因の明らかな問題がなければページ完了とする。
11. 作業記録、修正前後HTML、miChecker確認結果、分類、未解消理由をスプレッドシートに残す。

### Expected Benefits

- ページごとの文脈を確認しながら進められる。
- 作業者がAGENTの判断をその場で確認・修正できる。
- 判断に迷う箇所や要質問箇所を早い段階で検出しやすい。
- CMS管理画面の入力欄や表示結果を見ながら対応できる。
- 最終HTMLを出力できるため、CMSへの登録作業と証跡管理をつなげやすい。
- レンダリング結果を見ながら候補を処理できるため、ソースコードだけを見るより作業者が問題箇所を把握しやすい。

### Key Risks

- 1ページずつ対応するため、一括処理に比べて処理時間が長くなる可能性がある。
- 作業者の判断負荷が残りやすい。
- AGENTの提示内容や作業者の操作がページごとにばらつくと、品質の平準化が難しくなる可能性がある。
- 実行画面が検査結果を出すだけで終わると、作業者がCMS上で再修正する負荷が残る。
- 修正後HTMLの出力がCMS入力欄の制約と合わない場合、登録時に再調整が必要になる。
- レンダリング結果とHTMLソースの同期がずれると、採用した候補が最終HTMLに正しく反映されたか確認しづらくなる。

## Common Success Criteria

どちらのワークストリームでも、次の状態を目指す。

- 作業者が目視で探すアクセシビリティ問題の数を減らす。
- 修正内容、修正理由、関連ナレッジ、要確認事項が記録される。
- AGENTが機械的に対応できる範囲と、人間が確認すべき範囲が明確になる。
- CMS登録後の作業者確認と承認者確認を完全には省略せず、品質保証の工程として維持する。
- スプレッドシート上の作業記録・証跡管理と接続できる。
- miCheckerで本文コンテンツ起因の明らかな問題が残る場合は、修正、例外理由、エスカレーションのいずれかが記録されている。
- miCheckerを使う場合は、バージョン、実行日、検査対象、指摘分類、対応結果を証跡として残せる。
- Goal 2では、作業者が確認した最終HTMLを出力し、CMS登録に使える。
- Goal 2では、すべての修正候補が採用、編集して採用、却下、要確認のいずれかに分類されている。

## Comparison Point

今後の検討では、Goal 1とGoal 2を次の観点で比較する。

- 作業時間の短縮効果
- アクセシビリティ修正品質の安定性
- 誤変換や見落としの検出しやすさ
- CMS登録作業との相性
- miChecker指摘の解消・分類・証跡化のしやすさ
- Goal 2実行画面での問題箇所表示、候補採用、最終HTML出力のしやすさ
- レンダリングHTML上で候補を安全に採用/編集/却下できるか
- 作業者と承認者の負荷
- スプレッドシートでの証跡管理のしやすさ
- 自治体固有ルールへの対応しやすさ
