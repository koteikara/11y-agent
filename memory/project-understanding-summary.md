# Project Understanding Summary

## 目的

このプロジェクトは、公共団体向けCMSへのサイト移行と、移行時に行うアクセシビリティ修正を一体で効率化するための検討・実装・知識整理を行う。単なる旧サイトHTMLの再現ではなく、新CMSの入力構造に合わせてコンテンツ部分を再登録し、WCAG/JISおよび実務上重要なmiCheckerの品質シグナルを踏まえて、作業者・承認者が説明可能な形で確認できる状態を目指す。

## 対象範囲

主対象は、旧サイト全体ではなくCMSに登録する本文コンテンツである。ヘッダー、グローバルナビゲーション、パンくず、検索、フッター、広告、解析タグ、CMSテンプレート共通部品は原則として移行修正の対象外とし、外部検査ツールの指摘も `content`、`old-site-template`、`new-cms-template`、`unknown` に分類して扱う。

## 作業の中心課題

1. 旧サイトからページ一覧を作成し、テンプレートやサイトカテゴリーごとに整理する。
2. 旧ページ全体HTMLから、CMS登録対象となるコンテンツ部分を抽出する。
3. 抽出済みHTMLを、新CMSの構造・テンプレート・入力欄に合う形へ整える。
4. `a11y-migration-kb/` のルールに沿って、画像、リンク、表、見出し、テキスト表記、ファイルリンクなどのアクセシビリティ修正候補を作る。
5. 機械的に直せるもの、AIが下書きできるもの、人間確認が必要なもの、SV・顧客確認が必要なものを分ける。
6. 修正理由、判断根拠、miChecker結果、未解消理由、要確認事項を証跡として残す。

## 既存ナレッジの位置づけ

`a11y-migration-kb/` は、移行・アクセシビリティ修正の主基準である。1ルール1Markdownを基本とし、テキスト、リンク、画像、表、ファイル、HTML構造、CMS操作、質問プロトコル、WCAG/JIS根拠、miChecker逆引き情報を持つ。新しいルールや手順を追加する場合は、実作業者が判断に使える粒度で書き、判断に迷う内容や行政情報の意味を変える可能性がある内容は確定せず要確認として扱う。

## miCheckerの扱い

miCheckerは `a11y-migration-kb/` を置き換える主基準ではなく、公共団体案件で検収・品質確認に強く影響し得る受け入れシグナルとして扱う。本文コンテンツ起因の明らかなmiChecker問題が残る場合は原則完了扱いにせず、ページ全体検査で出たテンプレート起因の指摘は移行HTML修正とは分離して記録する。KB未対応のmiChecker指摘は、既存ルールへのタグ追記、新規ルール作成、本文スコープ外分類のいずれかへトリアージする。

## 現在のワークストリーム整理

### Goal 1: CMS登録前の一括最適化

ページ一覧・テンプレート分類・コンテンツ抽出後に、AGENTでアクセシビリティ修正を一括適用し、CMS登録前に最適化済みHTMLを用意する。多数ページの同種修正には向くが、誤変換が広がるリスクとページ文脈の見落としがある。

### Goal 2: CMS登録中の段階的最適化

作業者が1ページずつHTMLを入力し、問題箇所、理由、関連KBルール、修正候補を確認しながら、採用・編集して採用・却下・要確認を選ぶ実行画面で支援する。最終HTMLと証跡を出力し、CMS登録と承認者確認につなげる。現在 `goal2-app/` としてPoCが実装されている。

### Goal 3: 旧ページ全体HTMLからのコンテンツ抽出

旧ページURLまたは全体HTMLから、本文候補をヒューリスティックに抽出し、抽出根拠や除外領域を示したうえでGoal 2へ渡す前段機能である。現在 `goal2-app/public/goal3.html` と `goal3.js` としてPoCが実装されている。

## 自動化と人間確認の分担

- 機械処理: 明確なHTML構造変換、既知パターンの表・リンク・ファイル表示文言修正、候補の初期分類、証跡出力。
- AI下書き: 画像alt案、見出しや説明文の改善案、文脈を踏まえた表構造整理案。ただし確定は人間確認前提。
- 作業者確認: 原文の意味保持、CMS入力欄制約、自治体表記方針、候補の採否、miChecker指摘分類。
- 承認者・SV・顧客確認: 行政情報の意味が変わる可能性がある修正、自治体固有ルール、KB未対応指摘、未解消理由の妥当性。

## 今後の検討で重視する観点

- コンテンツ抽出条件と除外領域を後から説明できること。
- 修正前後差分、修正理由、関連WCAG/JIS、関連KBルールを証跡化できること。
- miChecker結果を本文起因とテンプレート起因に分け、本文起因の未解消指摘を放置しないこと。
- 作業者が迷う回数を減らしつつ、人間が判断すべき行政情報・自治体方針を自動確定しないこと。
- 既存スプレッドシート管理やCMS登録作業と接続し、現場の手戻りを減らすこと。
